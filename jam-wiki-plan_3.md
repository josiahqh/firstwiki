# Jam Wiki — Implementation Plan

## About the Group

A work-based music jam group with two core purposes:

1. **Bond over playing music** — a social and creative outlet for colleagues
2. **Improve musical literacy** — every session and song selection is an opportunity to explore and understand music more deeply

The group has approximately five members. Sessions involve live mixing of mic, MIDI, USB, acoustic, and plugged-in instruments, with tracking/recording gear present but not always the primary focus.

---

## Platform

- **GitHub Pages** with **Jekyll** for static site generation
- Recommended theme: **Just the Docs** — sidebar navigation, search, and wiki-style layout out of the box
- Navigation style: **nested sidebar**, maximum depth of 4. If any proposed structure would exceed depth 4, this must be flagged for a structural decision rather than implemented silently.
- All content written in **Markdown** with **YAML front matter**
- Dynamic index pages built using **Liquid templates** and **Jekyll collections**

## Jekyll Collections

Five collections will be declared in `_config.yml`, replacing manual page management
for the five most structured content types:

| Collection | Folder | Index page | Looped via |
|------------|--------|------------|------------|
| Repertoire | `_repertoire/` | `_repertoire/index.md` | `site.repertoire` |
| Sessions | `_sessions/` | `_sessions/index.md` | `site.sessions` |
| Members | `_members/` | `_members/index.md` | `site.members` |
| Gear | `_gear/` | `_gear/index.md` | `site.gear` |
| Concepts | `_concepts/` | `_concepts/index.md` | `site.concepts` |

Index pages inside collection folders carry `index: true` in their front matter
so they are excluded from the Liquid loop that builds each collection's content list:

```liquid
{% assign items = site.sessions | where_exp: "s", "s.index != true" %}
```

Each index page is otherwise fully dynamic — hardcoded only for its title, prose
introduction, column headers, and the Liquid template code itself. All rendered
content is pulled live from collection file front matter at build time.

**Note:** The `_repertoire/planning/` subfolder is unconventional for a Jekyll
collection. Non-song pages within `_repertoire/` (index, proposed, planning pages,
song-template) all carry `index: true` so the Liquid loop excludes them cleanly:

```liquid
{% assign songs = site.repertoire | where_exp: "s", "s.index != true" %}
```

The nested `planning/` pages should be tested early in implementation to confirm
they render correctly alongside collection items.

### Concepts Section
- **Index** — master list of all concepts, with links to songs in the repertoire that cover each one

## Song Page Template

Each song in `_repertoire/` will use a consistent template with the following structure:

### Front Matter Fields

```yaml
---
title: "Song Title"
artist: "Artist Name"
proposed_by: "Name"
date_proposed: YYYY-MM-DD
status: backlog # backlog | actively-learning | performance-ready | shelved
key: ""
tempo: 0
time_signature: "4/4"
genre: ""
difficulty: "" # beginner | intermediate | advanced
musical_focus: ""
musical_focus_secondary: ""
target_practice_date: YYYY-MM-DD
tags: []
---
```

### Page Sections

1. **Overview** — description, why we picked it, reference recording link
2. **Musical Focus** — primary and secondary concept, what we're exploring, what to listen for, further listening
3. **Key, Tempo & Structure** — song structure table, chord chart
4. **Part Assignments** — member/instrument/role table, per-member focus notes
5. **Session Log** — running notes per practice (what went well, what needs work, decisions, next steps)
6. **Arrangement Notes** — how the song has been adapted for the group's lineup
7. **Resources** — table of links (recording, chart, tab, backing track, tutorial)

## Musical Concept System

Each song is selected in part to illuminate a specific musical concept. This serves
the group's literacy goal and gives song selection a clear pedagogical purpose.
Concepts are a first-class Jekyll collection (`_concepts/`), making them queryable
just like repertoire items, members, or gear.

### Concept Front Matter Schema

```yaml
---
title: "Syncopation"
slug: syncopation
category: rhythm  # rhythm | harmony | melody | dynamics | form | texture
difficulty_range: "beginner–intermediate"
summary: "Notes landing between beats rather than on them."
---
```

### Song–Concept Linking

The `musical_focus` and `musical_focus_secondary` fields on each song must match
a concept's `slug` exactly. This allows Liquid to automatically query related songs
from within a concept page:

```liquid
{% assign related = site.repertoire | where_exp: "s", "s.index != true" | where: "musical_focus", page.slug %}
```

This means:
- Song pages display their concept with a link to the full concept page
- Concept pages automatically list every song in the repertoire that covers them
- No manual cross-referencing is needed — keeping song front matter accurate is sufficient
- The `_concepts/` collection acts as the **authoritative vocabulary** for musical focus values;
  a slug that has no concept page should not appear on a song

### Concepts Index

The `_concepts/index.md` page loops over `site.concepts` and groups entries by
`category`, giving a curriculum-style view of all rhythm concepts, all harmony
concepts, and so on. This makes it easy to spot gaps in the group's rotation.

### Example Concepts

| Slug | Title | Category |
|------|-------|----------|
| `syncopation` | Syncopation | Rhythm |
| `modal-harmony` | Modal Harmony | Harmony |
| `call-and-response` | Call and Response | Form |
| `groove-pocket` | Groove & Pocket | Rhythm |
| `dynamic-contrast` | Dynamic Contrast | Dynamics |
| `voice-leading` | Voice Leading | Harmony |

## Contributor Tooling

All five members will contribute to the wiki. Since contributors should not need to
know Jekyll, Liquid, or front matter conventions, a suite of **GitHub Actions workflows**
provides a simplified interface. Each workflow is triggered via GitHub's "Run workflow"
UI — a web form anyone with repository access can fill out. Outputs are generated as
pull requests for a maintainer to review and merge.

### Workflows

| Workflow file | Trigger | What it does |
|---------------|---------|--------------|
| `new-song.yml` | Manual form | Generates a correctly structured `_repertoire/` song file with all front matter pre-filled from form inputs; opens a PR |
| `new-concept.yml` | Manual form | Generates a `_concepts/` file from form inputs; opens a PR |
| `update-status.yml` | Manual form | Updates the `status` field on a named song file; opens a PR |
| `assign-parts.yml` | Manual form | Updates part assignments on a named song file; opens a PR |
| `sync-sessions.yml` | Issue close event | When a `session-log` issue is closed, generates or updates a `_sessions/` file from the issue body and comments; opens a PR |

### Contributor Access Model
- All five members have GitHub accounts and repository access
- Members submit changes via PRs — either through workflow forms or by editing pages directly on GitHub
- A designated maintainer (initially you) reviews and merges PRs
- Over time, merge rights can be extended to trusted contributors

## Session Logs as GitHub Issues

Each session is tracked as a **GitHub Issue** using a structured issue template
(`session-log.md`). This makes session logging naturally collaborative — any member
can open the issue before a session and all members can add comments during or after,
without touching any files.

### Session Issue Template Fields
- Date and location
- Attendees
- Songs practiced (linked to repertoire pages)
- What went well
- What needs work
- Decisions made
- Next session plan

### Promotion to Jekyll Collection
When a session issue is closed, the `sync-sessions.yml` workflow fires automatically.
It reads the issue body and comments via the GitHub API, assembles a `_sessions/` file
in the correct format, and opens a PR. Once merged, the session appears in the wiki's
session index.

Manual promotion is also always an option — a maintainer can create the `_sessions/`
file directly from the issue content if the automated output needs adjustment.

### Issue Labels
- `session-log` — identifies session issues for the sync workflow
- `open` / `closed` — naturally tracks whether a session has been promoted to the wiki

## Implementation Order (suggested)

1. **`_config.yml`** — theme, collections, navigation, permalinks
2. **Home page** — mission, quick links
3. **Song template** — `_repertoire/song-template.md`
4. **Repertoire index** — Liquid template with status-filtered sections
5. **Concepts index** — foundational for song selection going forward
6. **Reference section** — quick start and session procedure first, then gear and room setup
7. **Session issue template** — `.github/ISSUE_TEMPLATE/session-log.md`
8. **Contributor workflows** — `new-song.yml` first as the highest-value tool, then remaining workflows
9. **Remaining pages** — members, repertoire/planning, jam frameworks, schedule, etc.
